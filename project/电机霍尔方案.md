# Motor Hall Angle Calculation: Current Scheme & Improvement Proposals

## Problem Statement

In the current system, two paths provide angle information:

| Path | Modbus Register | Method | Accuracy |
|------|----------------|--------|----------|
| Path 1 (Angle) | 0x2731 | RPM integration | Systematic overshoot ~1-2deg |
| Path 2 (Pulse accumulator) | 0x3714+0x3715 | Raw pulse counting | Hardware-accurate |

Empirical evidence: after a full open stroke (-2deg -> 88deg stop), Path 1 reports 88deg (triggers limit stop), while Path 2 calculates 86.83deg from raw pulse count. Path 2 is the ground truth -- it counts every Hall edge unconditionally in the hardware ISR with microsecond precision.

The core question: **Why does Path 1 overshoot by 1-2deg when its RPM measurement (tested with a signal generator) is accurate in steady state?**

---

## Current Scheme: RPM-Integration

### Data Flow

```
Hall A/B EXTINT interrupts (hardware, microsecond-precise)
    |
    |-- hall_pulse_counts[0/1]++          (every edge, unconditional)
    |
    |-- pulse_counter++                   (edge filtered by 50uss~200ms interval check)
    |
    |-- interval_us = Timer6_Timebase_GetDelta() -> Timer6_DeltaToUs()
    |       |
    |       +-- interval_history[6]         (rolling buffer, last 6 intervals)
    |              |
    |              +-- avg_interval = sum(interval_history) / 6
    |                     |
    |                     +-- raw_rpm = 60,000,000 / (avg_interval x pulses_per_rev)
    |                            |
    |                            +-- rpm_window[6]   (rolling buffer, last 6 RPM samples)
    |                                   |
    |                                   +-- filtered_rpm = sum(rpm_window) / 6
    |
    +-- motor_hall_update() runs @ 10ms (rpm_timer)
           |
           +-- calculate_rpm -> update_rpm_filter -> perform_stall_detection
                  |
                  +-- inst->filtered_rpm  (updated every 10ms)
```

```
RTurn_UpdateAngle() runs @ 1ms (DeviceManager update cycle)
    |
    |-- fRpm = MotorHall_Device_Read()        -> reads cached_rpm -> filtered_rpm
    |       NOTE: cached_rpm only refreshes every 10ms, so 9/10 cycles use stale data
    |
    |-- fCurrentSpeed = fRpm x 360 / 1183.0 / 60.0   (deg/s from RPM)
    |
    |-- u32DeltaMs = tickTimer_GetCount() - u32LastAngleTime  (time compensation, correct)
    |
    +-- fCurrentAngle += fCurrentSpeed x (u32DeltaMs / 1000.0)   (integrate)
```

### Root Cause: Dual Moving-Average Filter During Transients

Two cascaded 6-sample moving-average filters:

1. **Pulse interval filter** (`interval_history[6]`): averages 6 raw pulse intervals -> `avg_interval`
2. **RPM filter** (`rpm_window[6]`): averages 6 RPM samples -> `filtered_rpm`

Effective window: roughly 36 pulse edges (~3 motor revolutions, ~90ms at 2000RPM).

**Why this causes overshoot:**

During **acceleration** (motor starting): the 6 most recent intervals in `interval_history` are all shorter than the long-term average. Since `RPM = const / avg_interval`, shorter intervals -> higher RPM estimate -> angle integrates faster than actual position.

During **deceleration** (approaching limit): the reverse happens physically (intervals get longer), but the moving average window still retains some short intervals from the cruise phase. `avg_interval` is shorter than actual -> RPM still elevated -> angle overshoots the physical limit.

During **steady state**: the 6-sample window accurately represents the true mean. Signal generator testing confirms RPM is correct at constant speed.

The 1-2deg overshoot accumulates almost entirely during acceleration and deceleration transients. Each transient lasts ~0.5-1.0 seconds, covering 15-30deg of movement at typical speeds.

## Overview of Improvement Proposals

| # | Name | Approach | Effort | Removes RPM? |
|---|------|----------|--------|--------------|
| Current | RPM-Integration | `angle += RPM x dt` (6+6 filter cascade) | -- | No |
| A | Larger RPM Window | Increase `RPM_WINDOW_SIZE` from 6 to 20-30 | Trivial | No |
| B | EWMA RPM Filter | Replace arithmetic mean with exponential weighting | Small | No |
| C | Pulse-Direct Angle | `angle = pulse_count / 12 / ratio x 360` | Medium | **Yes** |

---

## Proposal A: Larger RPM Window

**Change:** `RPM_WINDOW_SIZE` 6 -> 20 (or 6 -> 30)

**How it works:** A larger window contains samples from further back in time, reducing the influence of the most recent (shortest) intervals during acceleration. The filtered RPM responds more slowly, reducing transient overshoot.

### Pros
- One-line change (`#define RPM_WINDOW_SIZE 20`)
- No structural code changes
- Reduces acceleration/deceleration RPM bias

### Cons
- Does not eliminate the root cause -- still integrating RPM
- Direction change detection becomes slower (3x confirmation on a 3x larger window)
- Still subject to interval->RPM->angle conversion error
- Larger window = more lag during genuine speed changes
- Window size is a trade-off: too large -> sluggish; too small -> noisy
- Only masks the symptom, does not fix the root cause

---

## Proposal B: Exponential Weighted Moving Average (EWMA)

**Change:** Replace arithmetic mean in `update_rpm_filter()` with:
```c
new_filtered_rpm = alpha x new_raw_rpm + (1.0f - alpha) x old_filtered_rpm
```
Typical alpha = 0.2~0.3 for mechanical systems.

**How it works:** Recent samples have higher weight, but old samples are never fully discarded. This provides a smoother transition during speed changes, avoiding the "stale fast-sample" problem in the arithmetic window. The filter continuously adapts without discrete window boundaries.

### Pros
- Smaller code change than Proposal C (modify `update_rpm_filter` and potentially `calculate_average_interval`)
- No window-size tuning (just alpha)
- Better transient behavior than arithmetic mean
- Industry-standard for motor speed filtering

### Cons
- Still integrating RPM -> angle (root cause not eliminated)
- Optimal alpha depends on motor dynamics (requires tuning)
- Still has "RPM updates every 10ms, angle integrates every 1ms" staleness issue
- Deceleration behavior still imperfect: EWMA responds to deceleration, but after a delay
- Signal generator testing would show accurate RPM (same as current scheme)
- Only reduces, not eliminates, transient angle error

---

## Proposal C: Pulse-Direct Angle (Recommended)

**Change:** Replace RPM integration in `RTurn_UpdateAngle()` with direct pulse-to-angle conversion:

```c
// Instead of:
//   fCurrentAngle += fCurrentSpeed x (u32DeltaMs / 1000.0f);

// Use:
uint32_t total_pulses = MotorHall_Device_GetTotalPulseCount(pstcDev->stcConfig.u8MotorHallDevId);
fCurrentAngle = (float)total_pulses / 12.0f / fReductionRatio * 360.0f;
```

Or, for compatibility with the signed accumulator (+/- direction), use `g_s32HallPulseAccum` directly:

```c
// Direct from global accumulator (already direction-aware, 1ms update rate):
fCurrentAngle = (float)g_s32HallPulseAccum / 12.0f / fReductionRatio * 360.0f;
```

**How it works:** Eliminates the entire RPM->interval->average->v(1/avg)->integrate pipeline. Each Hall edge contributes exactly `360 / 12 / ratio = 0.025deg` to the angle. No filtering, no estimation, no transients. The angle is a simple linear function of the pulse count.

The existing angle calibration logic (overcurrent -> re-locate to limit angle, limit detection, EventBus publishing) remains unchanged. Only the source of the raw angle value changes.

### Pros
- **Eliminates root cause**: no RPM -> no RPM estimation error -> no transient bias
- Angle accuracy matches Path 2 (hardware pulse counter) exactly
- No tuning parameters (no window size, no alpha)
- Same precision in acceleration, deceleration, and steady state
- Angle resolution: 0.025deg per pulse (4x better than 0.1deg display resolution)
- Calibration and limit-stop logic preserved unchanged
- Direction-aware (via `g_s32HallPulseAccum` signed accumulation)

### Cons
- Requires refactoring `dev_rturn.c` angle update path
- Angle now depends on `reduction_ratio` accuracy (same as current scheme)
- If reduction ratio is wrong, angle is proportionally wrong (no longer self-correcting via RPM)
- Must handle the cumulative nature: angle can drift if `g_s32HallPulseAccum` is not periodically reset
- Existing reset-via-Modbus (write 0 to 0x3714) already provides this reset capability
- Requires careful testing of overcurrent re-calibration behavior

## Comparison Summary

| Criterion | Current | A: Large Window | B: EWMA | C: Pulse-Direct |
|-----------|---------|-----------------|---------|-----------------|
| Steady-state accuracy | Good (verified) | Good | Good | Excellent |
| Acceleration accuracy | Overshoots ~1deg | Reduced overshoot | Reduced overshoot | **No error** |
| Deceleration accuracy | Overshoots ~1deg | Reduced overshoot | Reduced overshoot | **No error** |
| Code change scope | -- | 1 line | ~20 lines | ~50 lines |
| Tuning required | Yes (window=6) | Yes (window=?) | Yes (alpha=?) | **No** |
| Matches Path 2 | No | No | No | **Yes** |
| Root cause addressed | -- | No | No | **Yes** |
| Risk | -- | Low | Low | Medium |

## Recommendation

**Proposal C (Pulse-Direct Angle)** is the recommended approach. It eliminates the root cause entirely by removing RPM from the angle calculation. Proposals A and B reduce the symptom but leave the fundamental problem intact.

The key insight: Hall sensors count discrete edges. Every edge is a fact. Converting edges -> intervals -> RPM -> angles introduces estimation error at every step. Using edges -> angles directly eliminates all estimation steps.

```

    |

```

## Why the Old Scheme Was Inaccurate

### Root Cause: Dual Moving-Average Filter During Transients

Two cascaded 6-sample arithmetic mean filters -- `interval_history[6]` -> `avg_interval`, then `rpm_window[6]` -> `filtered_rpm` -- create an effective window of approximately 36 Hall edges (~90ms at 2000 RPM).

**During acceleration** (motor starting, ~0.5s):

The moving average window fills with progressively shorter pulse intervals. Since `RPM = const / avg_interval`, an average biased toward recent (shorter) intervals produces an RPM estimate higher than the true instantaneous speed. Each 1ms integration cycle multiplies this inflated RPM by deltat, systematically accumulating excess angle.

**During deceleration** (approaching limit, ~0.5s):

Pulse intervals lengthen as the motor slows, but the sliding window retains several short intervals from the cruise phase. The average interval is shorter than actual -> RPM remains elevated -> angle overshoots the physical position by the time the limit triggers.

**In steady state** (constant speed, ~3-4s during stroke):

The 6-sample window accurately represents the true mean interval. Signal generator testing confirms RPM is correct at constant speed. No systematic error accumulates during this phase.

The net result: approximately 1-3deg of overshoot accumulated during acceleration and deceleration transients. The angle integration reaches 88deg before the motor physically covers the full 90deg stroke, causing premature limit triggering.

### Empirical Confirmation

Before the fix, a full open stroke (-2deg -> 88deg limit) produced:

| Source | Value | Interpretation |
|--------|-------|---------------|
| Path 1 (angle, 0x2731) | 88.00deg | Clamped at limit; triggers stop |
| Path 2 (pulses, 0x3714) | 3424 pulses = 86.83deg | Ground truth -- actually stopped short |

Path 2 reads **less** than 90deg because Path 1 reached 88deg prematurely and stopped the motor before the full stroke completed.

## Why the New Scheme Is Accurate

The pulse-direct formula eliminates all estimation steps:

```
fCurrentAngle = fMinAngle + g_s32HallPulseAccum x (360.0f / 12.0f / fReductionRatio)
```

Each term has defined error characteristics:

| Term | Error Source | Magnitude |
|------|-------------|-----------|
| `fMinAngle` | User calibration | +/-0.1deg (one-time, at close limit) |
| `g_s32HallPulseAccum` | +/-0 pulses (hardware ISR, every edge counted) | 0 |
| `360.0 / 12.0 / ratio` | Ratio accuracy | Proportional; same for both paths |
| deltat | Not used | Eliminated |
| RPM | Not used | Eliminated |
| Filter cascade | Not used | Eliminated |

The angle resolution is `360 / 12 / 1183 = 0.025deg` per pulse -- 4x finer than the 0.1deg display resolution.

Both Path 1 and Path 2 now derive from the same `g_s32HallPulseAccum` variable, guaranteeing they are consistent. There is no separate RPM integration path that can diverge.

### Forward Limit De-Clamp

Additionally, the forward limit clamp was removed:

```c
// Before: clamped -- real overshoot hidden
if (fCurrentAngle >= fMaxAngle) {
    fCurrentAngle = fMaxAngle;  // force 88.0deg
    // ... publish limit event
}

// After: unclamped -- real value preserved
if (fCurrentAngle >= fMaxAngle) {
    // ... publish limit event (stops motor)
    // angle continues to reflect pulse-direct calculation
}
```

The reverse (closing) limit clamp is preserved per product requirement -- the angle never displays below `fMinAngle` when closing.

## RPM Calculation: Pros and Cons

RPM continues to use the interval-based method for speed display and stall detection. This method has inherent trade-offs:

### Pros

- **Fast response at high speed**: at 2000 RPM, pulse interval is ~2500uss. Six samples capture RPM in ~15ms, providing responsive speed feedback during cruise.
- **Good steady-state accuracy**: when speed is constant, the 6-sample arithmetic mean is an unbiased estimator of the true mean interval. Validated with signal generator testing.
- **Simple implementation**: arithmetic mean requires no tuning parameters (unlike EWMA's alpha).

### Cons

- **Transient bias**: during acceleration/deceleration, the moving average lags behind true speed. The direction of bias depends on the recent trend in interval lengths.
- **Dual filter cascade**: the `interval_history -> avg_interval -> raw_rpm -> rpm_window -> filtered_rpm` pipeline compounds estimation error. Each averaging step adds its own lag.
- **Low-speed degradation**: at very low RPM, pulse intervals approach or exceed `MAX_PULSE_INTERVAL_US` (200ms), causing `is_running` to toggle and RPM to drop to zero spuriously. Stall detection (50ms timeout) can also interfere.
- **10ms staleness**: `filtered_rpm` updates every 10ms, but the angle previously consumed it every 1ms. Nine out of ten angle integration cycles used stale RPM data. (This is now irrelevant for angle, but the staleness still affects speed display responsiveness.)
- **Resolution at low speed**: at 100 RPM, only ~20 pulses arrive per second. The 6-sample window spans ~300ms, making the RPM reading sluggish during slow movements.

These RPM limitations are **acceptable** because RPM is used only for display and stall detection -- not for angle calculation. The accuracy requirements for RPM (~50 RPM resolution for a speed gauge) are far looser than for angle (~0.1deg for position control).

# 中文总结

## 旧方案为什么不准确

旧方案用 RPM 积分来算角度。RPM 本身在稳态下是准的（用信号发生器验证过），但角度不准，原因有两个：

1. **加减速时 RPM 虚高**：用了两层 6 样本滑动算术平均滤波。加速时，窗口里全是越来越短的脉冲间隔，算出来的 RPM 比实际快。减速时，窗口里还夹着匀速段的短间隔，RPM 依然偏高。RPM 偏高 -> 角度积得快 -> 提前到达 88 度限位 -> 电机提前停下。

2. **RPM 和角度解耦**：RPM 每 10ms 更新一次，但角度每 1ms 积分一次，9/10 的积分周期用的是过期数据。

实测证据：关窗到底 -> 重置脉冲 -> 开窗到 88 度自动停。Path 2（脉冲累计）显示约 3424 脉冲 = 86.83 度，Path 1（角度）显示 88 度。Path 2 更接近真实值，说明电机在物理转到 86.83 度时 Path 1 就已经算到 88 度而提前停了。

## 新方案为什么准确

方案 C：角度不再依赖 RPM，直接从脉冲累计换算。



每个霍尔边沿无条件在硬件中断里计数，每个脉冲精确对应 0.025 度。不需要估算间隔、不需要求 RPM、不需要滤波、不需要积分。Path 1 和 Path 2 现在走同一条数据源，不会再出现二者不一致的情况。

## RPM 为什么保留不动

RPM 仍然用原来的间隔测量 + 双滤波方案。虽然这套方案有加减速滞后的问题，但：

- 转速显示不需要角度那么高的精度，差几十 RPM 无所谓
- 堵转检测只需要判断 RPM 是否为 0，不依赖精度
- 信号发生器已验证稳态精度足够

角度要求 0.1 度的精度（控制限位停止），转速要求 10 RPM 的精度（显示用），两个场景的精度需求差了两个数量级。所以 RPM 可以保留原方案，角度必须改。

## 额外改动

开窗限位后不再压制角度显示。之前到了 88 度就把角度强制写为 88.00，即使实际超调到 89 度也看不见。现在去掉了这个打压，开窗超调多少一目了然。关窗方向保持压制不变，最低只显示 -2 度，这是客户要求的产品特性。
